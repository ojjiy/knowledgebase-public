【アップデート】 Amazon Aurora DSQL の Change Data Capture が一般提供になりました

Date: 2026-07-09

Source: rss / DevelopersIO

Original URL

Amazon Aurora DSQL の Change Data Capture (CDC) 機能が、プレビュー発表から約 2 ヶ月後の 2026 年 7 月 8 日に一般提供 (GA) となりました。この機能は、コミット済みのデータベース変更を Amazon Kinesis Data Streams へ準リアルタイムで配信するものです。 **主な特徴:** * **配信先:** Amazon Kinesis Data Streams (BYOT: Bring Your Own Target モデル) * **対象:** クラスター内の全ユーザーテーブルの行レベル変更 * **レコード形式:** 構造化 JSON * **配信保証:** 少なくとも 1 回 (at-least-once) * **順序:** UNORDERED (コミット順に近いが厳密な保証なし) * **マルチリージョン:** 1 リージョンの単一ストリームで全リージョンの変更をキャプチャ * **性能影響:** データベースワークロードへの影響なし **ユースケース:** * ダウンストリームシステムの同期 (検索インデックス、キャッシュ、データウェアハウス、分析システムへのレプリケーション) * イベント駆動アーキテクチャ (データベース変更をトリガーとしたワークフロー、通知、マイクロサービスの起動) * 監査証跡 (コンプライアンス、デバッグ、履歴分析) * プロデューサーとコンシューマーの疎結合化 (CQRS など) **プレビューからの変更点:** GA 版の最も重要な変更点は、CDC レコードの `op` フィールドで `UPDATE` が `c` (INSERT と区別不可) から `u` へと区別されるようになったことです。これにより、ダウンストリーム側で変更種別を正確に判別できるようになりました。 また、トランザクション内の複数操作を 1 レコードにまとめる「write-set compaction」の動作が明文化されました。同一トランザクション内で INSERT された行を UPDATE した場合、`op: "c"` の 1 レコードのみが発行され、`after` フィールドには最終状態が含まれます。INSERT された行を同一トランザクション内で DELETE した場合は、正味の変化がないためレコード自体が配信されません。 **注意点:** * 配信保証は `at-least-once` のため、重複レコードが発生する可能性があります。`source.ts_ns` と主キー値の組み合わせで重複を検出できます。 * 順序は `UNORDERED` なため、厳密な順序が必要な場合は `source.ts_ns` を使用して受信側で並べ替える必要があります。同一トランザクションのレコードグループ化には `source.txId` を使用します。 * 主キーのないテーブルでは、ダウンストリームでの重複排除や削除対象行の特定が困難になるため、CDC 対象テーブルには必ず主キーを定義する必要があります。 * Kinesis Data Streams の `MaxRecordSizeInKiB` はデフォルトの 1 MiB ではなく 10240 (10 MiB) を設定することが推奨されます。9 MiB を超えるレコードは分割されるため、受信側での再組み立てが必要です。 * 1 クラスターあたりの CDC ストリーム数には上限があります。 **料金:** CDC の料金は、キャプチャしたデータ量に基づいて StreamDPU として計測されます。StreamDPU は通常の DPU と同じレートで課金されます。変更がない場合は StreamDPU の課金はなく、Kinesis Data Streams 標準料金が別途発生します。 **まとめ:** Amazon Aurora DSQL の CDC 機能は、プレビュー時の課題が解消され、write-set compaction の動作も明確になったことで、イベント駆動アーキテクチャの起点として本番採用が現実的になりました。`at-least-once` および `UNORDERED` という特性を理解し、ダウンストリーム側で重複排除と順序付けを実装することで、検索インデックスの同期や監査証跡などのユースケースに幅広く活用できます。
