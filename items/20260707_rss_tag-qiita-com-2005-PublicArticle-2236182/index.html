【AgentCore】Runtime をパブリックにデプロイしたけど、勝手にアクセスされない？

Date: 2026-07-07

Source: rss / Qiita Trend

Original URL

Amazon Bedrock AgentCore Runtime をパブリックにデプロイしても、不正アクセスや不正利用はされないことが確認されました。 AgentCore Runtime のエンドポイント URL はインターネットからアクセス可能ですが、URL を知っているだけではエージェントを呼び出すことはできません。これは、以下の3層のセキュリティ機構によって保護されているためです。 1. **第1層：認証（IAM SigV4 署名 or JWT Bearer Token）** リクエストには、有効な AWS アクセスキーIDとシークレットアクセスキーから計算された暗号学的な署名（IAM SigV4）が含まれている必要があります。署名が正しくないリクエストは、エンドポイントに到達する前に拒否されます。 2. **第2層：認可（IDベース - IAM ポリシー）** 呼び出し元には、`bedrock-agentcore:InvokeAgentRuntime` 権限が付与された IAM ポリシーが必要です。この権限がない場合、アクセスは拒否されます。 3. **第3層：認可（リソースベース - Runtime 側のポリシー、任意）** さらに、Runtime にリソースベースポリシーをアタッチすることで、特定の IAM ロールなど、呼び出し元をより厳密に限定することができます。このポリシーにより、許可されたプリンシパル以外からのアクセスを明示的に拒否することが可能です。 **実測による検証結果:** * **認証ヘッダーなしでのアクセス:** 「Missing Authentication Token」として 403 Forbidden で拒否されました。 * **偽の認証情報でのアクセス:** 「Authorization method mismatch」として 403 Forbidden で拒否されました。これは、Runtime が SigV4 認証で構成されているため、Bearer トークン形式のリクエストが認証方式の不一致として処理されたためです。 * **正規の AWS 認証情報と IAM 権限でのアクセス:** 正常にレスポンスが返却され、エージェントが実行されました。 このように、AgentCore Runtime はデフォルトで強力な認証・認可メカニズムを備えており、パブリックエンドポイントであっても安全に利用できます。より厳密なアクセス制御を行いたい場合は、リソースベースポリシーの活用が推奨されます。また、AgentCore Runtime は JWT Bearer Token 認証もサポートしており、ユースケースに応じて選択可能です。
