# タイトル Securing the AI Agent: A Unified Framework for Multi-Layer Agent Red Teaming [View Paper](http://arxiv.org/abs/2606.31227v1) ## 1. 問題意識：解決したい問題はなにか 近年、AIエージェントの基盤となるオープンソースのインフラ（モデルサービングエンジン、エージェントプラットフォーム、Model Context Protocol (MCP) エコシステム、言語モデル自体など）が急速に発展していますが、それらを保護するためのセキュリティツールは追いついていません。特に、AIエージェントの攻撃対象領域は、インフラ、プロトコル/ツール、エージェントの振る舞い、モデルという複数のレイヤーにわたって存在し、単一の検出パラダイムでは全てのレイヤーを網羅したセキュリティ対策を講じることが困難です。このため、AIエージェントのサプライチェーン全体にわたる、包括的かつ効果的なセキュリティ脆弱性評価手法が求められています。 ## 2. 提案手法のアプローチ：どのようなアイデア・アプローチで問題を解決しようとしたか 本論文では、「AI-Infra-Guard」という、AIエージェントのレッドチーム編成を単一の観察に基づき体系化するオープンソースフレームワークを提案しています。その中心的なアイデアは、「攻撃対象領域はレイヤー（インフラ、プロトコル/ツール、エージェントの振る舞い、モデル）に階層化されており、単一の検出パラダイムでは全てのレイヤーに対応できない」という認識に基づき、各レイヤーに最適な検出パラダイムを「レイヤー-パラダイムマッチング」として割り当てることです。 具体的には、以下の4つのレイヤーそれぞれに最適な検出パラダイムを適用しています。 * **インフラストラクチャレイヤー (L1)**: 75以上のAIコンポーネントと1,400以上の脆弱性ルールに基づく決定的ルールマッチング。 * **プロトコル/ツールレイヤー (L2)**: MCPサーバーおよびエージェントスキルのLLM駆動型セマンティック監査。 * **エージェント行動レイヤー (L3)**: マルチターンのブラックボックスエージェントレッドチーム。 * **モデルレイヤー (L4)**: 26以上の攻撃オペレータと16のデータセットを用いたジャイルブレイクハーネス。 このフレームワークは、AIエージェントのスキル（機能拡張モジュール）のサプライチェーン監査も包括しており、オープンソースとして公開することで、レイヤー-パラダイムマッチングをエージェントセキュリティの実践的な基盤とし、コミュニティが共有して発展させられることを目指しています。 ## 3. 成果：提案手法により、何が達成できたのか AI-Infra-Guardは、AIエージェントの攻撃対象領域全体を網羅する、包括的なセキュリティ評価フレームワークを提供しました。具体的には、以下の成果が達成されました。 * **包括的な攻撃対象領域のカバー**: インフラ、プロトコル/ツール、エージェントの振る舞い、モデルという4つの主要レイヤーに加え、AIエージェントのサプライチェーンにおける新たな攻撃対象領域である「エージェントスキル」の監査も実現しました。これは、現存するオープンソースフレームワークでは唯一であると主張されています。 * **レイヤーごとの最適化された検出パラダイム**: 各レイヤーの特性（検出に必要な証拠の種類、コスト、複雑さなど）に合わせて、決定的ルールマッチング、LLM駆動型セマンティック監査、マルチターンブラックボックスレッドチーム、ジャイルブレイク評価といった最適な検出パラダイムを適用することで、効率的かつ効果的な脆弱性検出を可能にしました。 * **AIインフラストラクチャの特化された脆弱性検出**: 従来のツールでは見逃されがちな、AIソフトウェア特有の不規則なバージョン文字列や、認証されていないアクセス、認証情報・設定の漏洩といった脆弱性を発見するための、AIコンポーネントに特化したフィンガープリントおよび脆弱性ルールセットを構築しました。 * **LLMをセキュリティ監査に活用**: LLMを単なる分類器としてではなく、コードの読解、ツールの利用、動的な動作確認を可能にする「ドメインセキュリティ監査官」として活用するハーネスを開発しました。これにより、静的解析だけでは発見できない、プロトコルレイヤーやエージェント行動レイヤーの複雑な脆弱性を検出できるようになりました。 * **AIスキルサプライチェーンのセキュリティ強化**: エージェントスキルという、AIエージェントの能力を拡張する重要なコンポーネントのサプライチェーンにおける、悪意のあるコードの混入や指示の隠蔽といったリスクを検出する専用の監査パイプラインを開発しました。 * **拡張性とコミュニティへの貢献**: AI-Infra-Guardをオープンソースで公開することで、コミュニティがこのフレームワークを基盤として、新たな検出モジュールや攻撃手法を開発・統合し、AIエージェントのセキュリティ全体を継続的に向上させていくための基盤を提供しました。 ## 4. 課題：本手法の限界・課題にはなにか。本文中で著者が言及しているものと、あなたが想定するもの ### 本文中で著者が言及している課題 * **LLMのオープンエンド性**: LLMベースの監査モジュールは、その表現力が高い反面、オープンエンドな性質から誤検出（過剰報告）が発生しやすい傾向があります。これを防ぐため、「Prompt-as-Rule」パラダイムで検出基準だけでなく、明示的な除外条件をエンコードすることが重要視されています。 * **LLM監査ツール自体の脆弱性**: LLM監査ツールが、分析対象のコードやツールの出力を処理する際に、それ自体が攻撃対象となり得ます。悪意のある指示を埋め込まれたデータやコードによって、監査計画が変更されたり、検出が抑制されたりするリスクがあり、これに対する防御策（プロンプトレベル、実行レベルでの対策）が講じられています。 * **検出手法のコストと時間**: 特にLLM駆動型監査やブラックボックスレッドチームは、決定的ルールマッチングに比べて計算コストや時間がかかります。このため、調査範囲の最適化（例: 機能認識、エスカレーション、早期停止ルール）によって、限られた予算を有効に使う工夫がされています。 * **最新のAIエコシステムへの追随**: AIエージェントのエコシステムは急速に進化しており、新たな攻撃手法や脆弱性が次々と出現します。AI-Infra-Guardは継続的な知識ベースの拡充やモジュールの拡張を予定していますが、この進化のスピードに常に追随していくことは継続的な課題となります。 ### あなたが想定する課題 * **LLMの「ブラックボックス性」と説明責任**: LLMが検出した脆弱性について、その判断根拠（証拠連鎖）を明確に提示しようとしていますが、LLM自体の内部的な判断プロセスは依然としてブラックボックス的です。より複雑な脆弱性や、微妙な判断が求められるケースにおいて、最終的な責任を誰がどのように負うのか、という点が運用上課題となる可能性があります。 * **「レイヤー-パラダイムマッチング」の完全性**: 提案されているレイヤー-パラダイムマッチングは、現状で考えられる最適な組み合わせですが、将来的にAIエージェントのアーキテクチャがさらに進化した場合、新たなレイヤーが出現したり、既存のレイヤーの性質が変化したりする可能性があります。その際に、このマッチング手法が常に有効であり続けるかどうかの検証が必要です。 * **誤検知と過少検知のバランス**: 各レイヤーで異なる検出パラダイムが採用されていますが、それぞれに誤検知（False Positive）と過少検知（False Negative）のトレードオフが存在します。特に、決定的ルールマッチングにおける「見逃しを避けるためのカバレッジ重視」の姿勢は、誤検知を増やす可能性があります。LLMベースのモジュールも、その学習データやプロンプト設計によって、これらのバランスが変化する可能性があります。 * **AIエージェントの実行環境の多様性への対応**: AI-Infra-Guardは、Dify、Coze、HTTP、WebSocket、標準モデルAPIなど、多様なプラットフォームに対応するためのアダプターを用意していますが、今後さらに多様化・複雑化するAIエージェントの実行環境（例: 分散型エージェント、ハードウェアアクセラレーションを利用したエージェントなど）に対して、迅速かつ効果的に対応できるかどうかが課題となります。 * **LLMのバイアスや安全性への影響**: LLM自体が持つバイアスや、学習データに起因する安全性の問題が、検出結果や監査プロセスの公平性・信頼性に影響を与える可能性があります。例えば、特定の種類の脆弱性に対して過剰に敏感になったり、逆に特定の攻撃手法を見逃したりする可能性があります。 ## 5. 技術詳細：提案手法の技術的な詳細について、できるだけ詳しく。数式やアルゴリズム、モデル構造なども可能な限り詳しく AI-Infra-Guardは、4つの検出モジュール（レイヤー）と、それらを統合・実行するオーケストレーション層から構成されています。 ### 検出モジュール (Detection Modules) #### M1: Infrastructure Layer (インフラストラクチャレイヤー) * **目的**: AIソフトウェアが実行されている環境をスキャンし、既知の脆弱性を持つコンポーネントを特定する。 * **パラダイム**: **決定的ルールマッチング (Deterministic Rule Matching)** * **AIコンポーネントの特定**: * **フィンガープリントルール**: サイトのHTMLボディ、アイコン（Faviconハッシュ）、HTTPヘッダーなどの特徴を正規表現やカスタム言語で記述されたルールに基づいてマッチングします。 * **ルール言語**: 小型ブーリアン式言語（`||` (OR), `&&` (AND), `! `(NOT), `()`, `==` (等価), `!=` (非等価), `contains` (部分文字列), `starts_with`, `ends_with`, `regex` (正規表現)）。 * **構文例**: `body.contains("<title>Dify</title>") || icon.hash == "97378986"` * **バージョン抽出**: コンポーネント特定後、別途定義されたルール（主に正規表現）を用いてバージョン情報を抽出します。バージョン文字列の正規化（例: `alpha` -> `0.0.0-alpha`）とバージョンライブラリとの比較により、不規則なバージョン形式に対応します。 * **脆弱性マッチング**: * **脆弱性ルール**: 特定されたコンポーネントとバージョンが、既知の脆弱性（CVEなど）に該当するかを判定します。 * **ルールファミリー**: 107のフィンガープリントルールと1,443の脆弱性ルール（75コンポーネント対象）が存在します。 * **証拠の強さ**: * **署名証拠 (Signature Evidence)**: * **Verified Exposures**: 実際に脆弱なパスをプローブし、機密情報（設定ファイル、認証情報など）が含まれていることを確認した場合。例: `/.env` へのアクセスで `.env` ファイルの内容が返却される。 * **Version-Based Findings**: 特定されたコンポーネントとバージョンが、公知の脆弱性アドバイザリに合致する場合。 * **Inferred Findings**: バージョンが抽出できない、あるいはアドバイザリがバージョンに依存しない場合に、コンポーネントの存在のみに基づいてリスクを報告する場合。 * **アーキテクチャ**: * **スキャニングエンジン**: Go言語で実装。 * **ターゲット管理**: 大規模CIDRブロックを扱うため、メモリ使用量を抑えるために、インメモリ/ディスクマップのハイブリッド構造を採用し、ストリームとしてイテレートします。 * **並行処理**: Sized WaitGroupでアクティブなプローブ数を制限し、Rate Limiterでスループットを制御します。 * **出力**: セキュリティスコア、アセットインベントリ、脆弱性詳細、ビジネスインパクト。 #### M2: Protocol/Tool Layer (プロトコル/ツールレイヤー) * **目的**: Model Context Protocol (MCP) サーバーやエージェントスキルにおける、ネットワーク経由で実行可能な脆弱性（コマンドインジェクション、情報漏洩など）を検出する。 * **パラダイム**: **LLM駆動型セマンティック監査 (LLM-Driven Semantic Auditing)** * **LLMハーネス**: * **構造**: LLMを単体ではなく、タスク構造、ツールセット、制御機構（Reason-Act Loop）、出力契約を持つ「ドメインセキュリティ監査官」として機能させるためのフレームワーク。 * **Reason-Act Loop**: 1. **Reason**: 現在の履歴に基づき、次に何を調査すべきか判断。 2. **Act**: ツールを呼び出す、またはコードを検索する。 3. **Observe**: ツールの結果やコードの応答を履歴に追加。 4. **Repeat**: 規定のイテレーション数に達するか、タスク完了まで繰り返す。 * **コンテキスト圧縮**: 大規模リポジトリを扱うため、コンテキストウィンドウを超えないように、古い履歴を要約して保存します。 * **ツールセット**: * **静的モード**: ファイル読み込み、シェル実行（grepなど）、コード検索、補完ツール。 * **動的モード**: MCPクライアントツール（enumerate tools, enumerate prompts and resources, call remote toolなど）。 * **モデル分割**: 役割（プライマリ推論、コード特化、補助）に応じて異なるLLMを使い分けることで、コストと精度のバランスを取ります。 * **モード**: * **静的監査 (Static Auditing)**: 1. **情報収集**: プロジェクト構造、依存関係などを把握。 2. **コード監査**: LLMがコードを読み込み、脆弱性パターンを検出。 3. **脆弱性レビュー**: 検出された候補を検証し、誤検出を除去。 * **検出パターン**: OWASP MCP Top 10に準拠（認証バイパス、コマンドインジェクション、ハードコーディングされた秘密情報、間接プロンプトインジェクションなど）。 * **「Prompt-as-Rule」**: 検出基準、高リスクコードパターン、除外条件を自然言語で定義。 * **動的監査 (Dynamic Auditing)**: 1. **情報収集**: MCPサーバーのメタデータ（ツール名、説明、スキーマ）を取得。 2. **悪性挙動テスト**: ツールを実際に呼び出し、意図しない挙動を試す。 3. **脆弱性テスト**: 特定された脆弱性に対してテストペイロードを送信。 4. **レビュー**: 検出された脆弱性を検証。 * **攻撃ベクター**: ツールメタデータ（説明文）自体が攻撃対象となる。 * **ペイロード生成**: LLMがテストペイロードを生成し、ツールの出力を分析して脆弱性を判断。 * **セキュリティ**: * **パスサンドボックス**: 監査ツール自体が、対象ディレクトリ外のファイルにアクセスできないように制限。 * **間接プロンプトインジェクション対策**: LLMへの指示と、ツールからの応答データを構造的に分離。 * **出力**: セキュリティスコア、リスク詳細、修正提案。 #### M3: Agent Behavior Layer (エージェント行動レイヤー) * **目的**: 稼働中のAIエージェント（アシスタント、チャットボットなど）が、ランタイムで示す行動の脆弱性を発見する。 * **パラダイム**: **マルチターンブラックボックスレッドチーム (Multi-turn Black-box Red Teaming)** * **アプローチ**: LLMエージェントが、対話を通じてターゲットエージェントの脆弱性を探索する。 * **Reason-Act Loop**: 1. **Reconnaissance**: ターゲットエージェントの能力（利用可能なツール、プロンプトなど）をプロファイリング。 2. **Detection**: 攻撃を仕掛ける。 3. **Review**: 発見された脆弱性を標準化された分類（OWASP Top 10 for Agentic Applications）にマッピング。 * **攻撃ツールの限界**: インターフェースは対話のみ。攻撃ツールは、ターゲットエージェントへのプロンプト送信（Primitive）と、その応答の受信。 * **シングルターン vs マルチターン**: * **シングルターン**: 1メッセージ、1応答、1リトライ。 * **マルチターン**: 攻撃エージェントのReason-Act Loopにより、対話履歴を考慮した連続的な攻撃（コンテキスト維持、スキル内エスカレーション、セッション継続）。 * **攻撃ファミリー**: データ漏洩、ツール乱用、間接インジェクション、認可バイパス。 * **「Prompt-as-Rule」（Manifest）**: 各攻撃ファミリーに対し、検出条件、確認方法、プローブ、エスカレーション手順、脆弱性判断基準を定義したプレイブック。 * **コスト意識**: * **Capability Awareness**: ターゲットが持たない機能を試さない。 * **Escalation**: 直接的なプローブから、エンコーディング、ロールプレイなどの回避手法へ段階的にエスカレート。 * **Stop Rules**: 脆弱性が確認されたら、それ以上のプローブを停止。 * **対話回数**: コスト指標として記録。 * **証拠の客観性**: * **SSRF Canary Token**: 制御したURLにアクセスさせ、埋め込んだトークンが返ってくるかで検証。 * **Planted Markers**: 隠し指示を埋め込んだコンテンツを処理させ、意図した応答（マーカーの出力など）で検証。 * **防御策**: * **Indirect Injection Defense**: LLMが分析対象のデータ（ツール応答、ファイル内容）を指示として解釈しないようにする。 * **Path Sandbox**: 監査ツールが対象ディレクトリ外にアクセスできないようにする。 * **Input Analysis vs Output Analysis**: ツールが自身で攻撃を仕掛ける場合でも、その「結果」に証拠を求める。 * **出力**: プロジェクト概要、セキュリティスコア、脆弱性リスト（対話履歴付き）。 #### M4: Model Layer (モデルレイヤー) * **目的**: AIモデル自体の安全性の整合性（Safety Alignment）が、敵対的なプロンプトに対してどの程度堅牢かを評価する。 * **パラダイム**: **大規模敵対的攻撃列挙 (Large-scale Adversarial Attack Enumeration)** - ジャイルブレイク評価 * **ジャイルブレイク評価ハーネス (Jailbreak Evaluation Harness)**: * **構成**: 攻撃生成モデル (Attacker)、ターゲットモデル (Target)、判定モデル (Judge) の3つの役割。 * **成分**: 攻撃オペレータ (Operators)、データセット (Datasets)、メトリクス (Metrics)。 * **攻撃オペレータ**: * **アルゴリズムオペレータ**: 決定的文字列変換（Base64、Vigenère暗号など）。 * **モデル駆動オペレータ**: LLM（Simulator）が攻撃を生成・駆動（ロールプレイ、コンテキスト強制、ツリー探索など）。 * **エンコーディング・難読化**: レキシカル分割、構造的隠蔽、順序・テンプレート操作など（70種類以上）。 * **マルチターンオペレータ**: Crescendo、Tree-of-Attacks、Best-of-Nなど。 * **データセット**: 16種類のデータセット（約7,000の有害プロンプト）。 * **判定モデル (Judge)**: LLMが担当。汎用ジャイルブレイク判定に加え、毒性、バイアス、偽情報、個人情報漏洩、エージェント固有のチェックなど、ハーム固有の判定モデルも用意。 * **評価プロセス**: 攻撃オペレータがプロンプトを変換・生成し、ターゲットモデルに入力。Judgeがその応答を評価し、成功/失敗を判定。 * **成功指標**: Attack-Success-Rate (ASR)。 * **出力**: 安全性評価（ASR）、成功したジャイルブレイクプロンプト、その応答。 #### Agent Skill Supply Chain Auditing (エージェントスキル サプライチェーン監査) * **目的**: AIエージェントの機能拡張モジュールである「スキル」に潜むサプライチェーンリスクを検出する。 * **パラダイム**: **LLM駆動型セマンティック監査 (LLM-Driven Semantic Auditing)** (M2の技術を応用) * **アプローチ**: スキルパッケージ全体を解析し、宣言された機能と実装された挙動の一致、隠し指示、不正な振る舞いを評価。 * **主要な質問**: 1. スキルが宣言する機能は何か？ 2. インストールスクリプト、コード、依存関係、設定は宣言された目的に一致するか？ 3. 隠された指示や不正な挙動は存在するか？ 4. インストールされた場合の最悪のセキュリティ影響は？ 5. 成果物全体は悪意があると分類されるべきか？ * **監査対象**: インストールスクリプト、コード、依存関係宣言、外部ダウンロードソース、圧縮アーカイブ、設定ファイルなど。 * **プロセス**: 1. **前処理**: ディレクトリ構造解析、エントリポイント、インストールスクリプト、依存関係などを特定。 2. **静的取得**: 危険なダウンロード、クラウドメタデータアクセス、不正なファイル読み込みなどを迅速に検出（ただし、これらは検出のヒントとして扱われる）。 3. **コンテキスト監査**: LLMがスキルパッケージ全体を解析。ツール環境内でディレクトリを移動し、ファイル読み込み、パターン検索、難読化解除、宣言された目的との比較を行う。 * **制限**: スキル自体は実行せず、監査ツール自体が安全に動作するための厳格な実行境界（ファイルアクセス制限、ツール呼び出し制限、出力サイズ制限など）を設ける。 * **外部連携**: Tencent Cloudの脅威インテリジェンスAPIと連携し、URLやハッシュの評判を評価。 * **出力**: リスク分類（Benign, Risky, Malicious）、証拠の場所、影響ファイル、説明、リスクカテゴリ、深刻度、推奨される緩和策。 ### オーケストレーション層 (Orchestration Layer) * **目的**: 上記の検出モジュールを統合し、スキャンリクエストの管理、ワーカーエージェントへのタスク割り当て、結果の収集・ストリーミングを行う。 * **アーキテクチャ**: * **バックエンド**: Gin Web Server、プラグイン管理、ストレージ、LLMプロバイダー。 * **ワーカーエージェント**: Goベースのワーカー（インフラスキャナーをインプロセスで実行）とPythonサブプロセス（LLM駆動モジュールを実行）。 * **タスクディスパッチ**: ロックフリー、アトミックインクリメントされたタスクIDにより、高速なGoスキャンと長時間のLLM監査を統一的なメカニズムで処理。 * **リアルタイムストリーミング**: 実行中のイベント（計画ステップ、ステータス更新、ツール使用、ログ、結果更新）をWebSocket経由でクライアントに即時ストリーミング。 * **知識ベース**: Fingerprints, CVEs, MCP Plugins, Jailbreak DatasetsなどをローカルファイルまたはサーバーAPIからロード。 * **デプロイメント形態**: * **集中型サーバー-エージェント**: CLIまたはWebサービスとして実行。 * **エージェントスキル**: 各種ホストエージェントがインストール・呼び出し可能なスキルとして提供。 * **プラグインシステム**: 攻撃オペレータ、脆弱性、メトリクスをプラグインとして動的にロード・拡張可能。 ## 6. この論文を140字以内のツイートで要約すると？
