【小ネタ】AgentCore Gatewayのインターセプターにカスタムヘッダーを渡してみた

Date: 2026-07-09

Source: rss / DevelopersIO

Original URL

Amazon Bedrock AgentCore Gatewayのインターセプター機能について、カスタムHTTPヘッダーがインターセプターに到達するかどうかが検証されました。 **結論として、カスタムヘッダーは特定の条件下でインターセプターに送信可能であることが確認されました。** **主要なポイント:** * **`passRequestHeaders` の有効化が必須:** * Gatewayの設定で `inputConfiguration.passRequestHeaders` を `true` にした場合のみ、リクエストヘッダーが `gatewayRequest.headers` に渡されます。 * この設定がされていない場合、`headers` は空のマップ `{}` となり、カスタムヘッダーは一切送信されません。 * **ヘッダーの小文字化:** * インターセプターに到達したカスタムヘッダーは、元のケースに関わらず小文字に正規化されていました（例: `X-Dept-Id` → `x-dept-id`）。 * ヘッダーを参照する際は、小文字で扱うことが推奨されます。 * **`Authorization` ヘッダーの扱い:** * `passRequestHeaders` を `true` にすると、`Authorization` ヘッダー（Bearerトークンなど）もそのまま渡されます。 * ログ出力時に機密情報が含まれる可能性があるため、注意が必要です。 * **セキュリティ上の注意点:** * カスタムヘッダーはクライアントによって書き換え可能であるため、認可などの重要な制御に直接使用することは推奨されません。 * ヘッダーの改ざんがないことが保証されている（例: 信頼できるプロキシによる付与・上書き）構成の場合に限り、検討の余地があります。 **検証環境と方法:** * Amazon Cognito User Pool（client\_credentialsフロー）を用いたインバウンド認証。 * Python 3.13のLambda関数をREQUESTインターセプターとして使用。 * `passRequestHeaders` を `true` に設定したGatewayと、設定しない場合の両方で比較検証。 * カスタムヘッダー（`X-Dept-Id`, `X-App-Id`, `X-Trace-Note`）を付与してツールを呼び出し、インターセプターLambdaのCloudWatch Logsで受信内容を確認。 * ヘッダー値による簡易的なアクセス制御（`X-Dept-Id` の許可リスト）も技術的に可能であることが実証されました。 **補足事項:** * REQUESTインターセプターは、リクエストボディ（JSON-RPCリクエスト）の参照・加工も可能です。これにより、ツールの引数などを動的に変更したり、条件分岐させたりできます。 * `transformedGatewayResponse` を返すことで、ターゲットのLambdaを呼び出さずにインターセプターから直接クライアントに応答を返すことができます。 この検証により、AgentCore Gatewayのインターセプターでカスタムヘッダーを扱うための条件と注意点が明確になりました。`passRequestHeaders` の設定を適切に行うことで、カスタムヘッダーを利用したログ記録、監査、あるいは限定的な情報付与などが可能になります。
