[小ネタ] Bitnami WordPress AMI は PHP などの同梱コンポーネントをアップグレードできない

Date: 2026-07-07

Source: rss / DevelopersIO

Original URL

Bitnami WordPress AMI を利用して EC2 上に構築された WordPress サーバーにおいて、PHP のバージョンを 7.4 から 8.x へアップグレードしようとしたところ、Bitnami の構成では PHP 単体のアップグレードが不可能であることが判明しました。 Bitnami イメージは、Apache/NGINX、MariaDB、PHP などの主要コンポーネントを `/opt/bitnami` 配下に独自ビルドで同梱しており、これらのコンポーネントは一体化して依存関係が構築されています。そのため、Bitnami 公式は GitHub 上で、これらのベースコンポーネントの個別アップグレードは不可能であると明言しています。PHP だけを差し替える公式な手順は提供されておらず、推奨される対応は、新しいバージョンの Bitnami AMI を使用してサーバーを再構築し、既存のデータを移行するという方法です。 さらに、Debian 10 (buster) を OS として使用している場合、Bitnami 同梱の PHP を無効化して OS のパッケージマネージャー (Sury リポジトリなど) から PHP をインストールするという代替手段も、Debian 10 の LTS サポート終了 (2024 年 6 月末) に伴い、Sury リポジトリでの Debian 10 向け PHP パッケージの配布終了により、利用できなくなっています。 したがって、古いバージョンの OS を使用している Bitnami サーバーでは、非公式な方法も含めて PHP をアップグレードする手段が事実上存在しない状況です。技術的には PHP をソースコードから自前でビルドして導入することも可能ですが、その後のセキュリティ更新のたびに再ビルドが必要になり、動作保証も得られないため現実的ではありません。 また、Bitnami の Marketplace で希望する PHP バージョンの公式 AMI を探すことも試みましたが、必要なバージョンが見つからず、「希望する PHP バージョンの Bitnami AMI に載せ替える」という選択肢も取れませんでした。 最終的に、この事象への対応としては、最新の Bitnami WordPress AMI (PHP 8.x) を使用して新しい EC2 インスタンスを構築し、WordPress ファイルとデータベースを新サーバーへ移行しました。新サーバーで動作検証を行った後、DNS の A レコードを新サーバーの Elastic IP へ切り替え、その後 SSL 証明書を取得・反映させるという一連の移行作業を実施しました。 Bitnami 構成の WordPress サーバーでは、PHP のサポート期限切れがそのままサーバー全体の移行案件につながることを示唆しており、PHP のサポート期限を確認し、データ移行、検証、DNS 切り替えまで含めた計画的な移行準備が重要であると結論付けています。
